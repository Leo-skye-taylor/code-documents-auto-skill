# 代码文档自动管理技能

面向 AI 的代码文档管理系统。

## 核心原则

1. **AI 优先文档**：所有文档都为 AI 消费和项目接管而结构化
2. **强制读取**：任何开发前，必须读取相关文档
3. **完整审计**：所有变更都记录完整上下文和决策理由
4. **活文档**：文档随代码库演进，永不陈旧
5. **全自动**：所有操作由 AI 执行，无需用户手动操作

## 自动初始化

**当 AI 第一次接触项目时，必须自动执行：**

1. **检查项目状态**
   - 检查是否存在 `.ai-context/` 目录
   - 检查是否存在 `CLAUDE.md` 文件
   - 检查是否存在 `AGENTS.md` 文件

2. **自动设置规则**
   - 如果 `CLAUDE.md` 不存在，创建并追加强制规则
   - 如果 `CLAUDE.md` 存在但缺少规则，追加规则
   - 如果 `AGENTS.md` 不存在，创建并追加 Agent 规则
   - 如果 `AGENTS.md` 存在但缺少规则，追加规则

3. **自动扫描代码库**
   - 如果 `.ai-context/` 不存在，运行 `/docs-scan` 生成文档

4. **自动初始化工作流**
   - 确保工作流日志存在
   - 确保读取日志存在

**用户不需要执行任何手动操作，AI 会自动完成所有初始化。**

## 强制工作流（全自动）

**重要**：此工作流是强制性的，AI 必须自动执行，用户无需手动操作。

### 用户使用方式

用户只需要正常对话，例如：
- "帮我修改登录功能，添加记住密码"
- "修复认证模块的 bug"
- "重构数据库层"

**AI 会自动完成所有文档相关操作。**

### 代码变更前（AI 自动执行）

当用户请求修改代码时，AI 必须自动执行以下操作：

1. **自动识别影响范围**
   - 分析要修改的文件
   - 确定涉及的模块
   - 确定涉及的功能

2. **自动读取相关文档**
   - AI 自动查找并读取所有相关文档
   - 包括：架构文档、编码规范、模块文档、功能文档、API 文档
   - 读取会被自动记录和验证

3. **自动启动工作流**
   - AI 自动执行：`bash ~/.claude/skills/code-documents-auto-skill/scripts/enforce-workflow.sh start -f "文件列表" -m "模块列表"`
   - 工作流会话自动创建

4. **自动验证读取完成**
   - AI 自动执行：`bash ~/.claude/skills/code-documents-auto-skill/scripts/enforce-workflow.sh check -f "文件列表"`
   - 验证通过后才能编辑代码

### 代码变更后（AI 自动执行）

当代码修改完成后，AI 必须自动执行以下操作：

1. **自动记录变更**
   - AI 自动执行：`bash ~/.claude/skills/code-documents-auto-skill/scripts/record-changes.sh "变更描述"`
   - 生成完整的变更记录

2. **自动更新文档**
   - 自动更新模块文档
   - 自动更新功能文档
   - 自动更新 API 文档
   - 自动生成变更记录

3. **自动完成工作流**
   - AI 自动执行：`bash ~/.claude/skills/code-documents-auto-skill/scripts/enforce-workflow.sh complete -f "修改的文件列表"`
   - 工作流会话自动完成

### 强制机制

工作流通过以下方式强制执行：
1. **工作流会话**：必须有活跃会话才能编辑
2. **读取验证**：必须读取所有相关文档
3. **文档生成**：变更后必须生成/更新文档
4. **日志记录**：所有操作都记录用于审计
5. **Hooks 拦截**：如果 AI 跳过步骤，hooks 会阻止操作并提示

**警告**：跳过任何步骤将导致工作流失败，hooks 会阻止代码编辑。

## 命令

### `/docs-scan` - 初始代码库扫描

对整个代码库执行全面扫描：

1. **文件发现**
   - 扫描所有源代码文件
   - 识别配置文件
   - 查找现有文档
   - 映射项目结构

2. **分析**
   - 代码架构和模式
   - 依赖和关系
   - 模块边界和接口
   - 数据流和控制流

3. **文档生成**
   - 创建层次化文档结构
   - 生成模块文档
   - 映射功能实现
   - 记录 API 和接口

**用法：**
```
/docs-scan [目录]
```

**输出：**
- `.ai-context/` 目录包含结构化文档
- `.ai-context/README.md` - 项目概览
- `.ai-context/modules/` - 模块文档
- `.ai-context/features/` - 功能文档
- `.ai-context/api/` - API 文档
- `.ai-context/architecture.md` - 架构概览

### `/docs-read` - 开发前读取

**强制**在任何开发工作前执行，不可跳过。

1. **自动文档发现**
   - AI 分析要修改的文件/模块
   - 自动查找所有相关文档
   - 包括：模块文档、功能文档、API 文档、架构、规范

2. **强制读取**
   - 所有相关文档必须读取
   - 读取会被记录和验证
   - 所有文档读取前不能继续

3. **工作流会话**
   - 启动工作流会话
   - 跟踪所有读取活动
   - 启用开发后文档生成

**用法：**
```
/docs-read [文件或模块]
```

**示例：**
```bash
# 读取特定文件的文档
/docs-read src/auth/login.ts src/user/user.service.ts

# 读取整个模块的文档
/docs-read auth,user

# 读取功能的文档
/docs-read login-feature
```

**强制机制：**
- 未读取所有文档不能开发
- 读取被记录，不能伪造
- 工作流会话必须活跃
- 编辑前验证必须通过

**读取内容：**
- 模块文档（受影响模块）
- 功能文档（受影响功能）
- API 文档（受影响 API）
- 架构概览
- 编码规范
- 相关决策记录

### `/docs-record` - 变更记录

**强制**在任何开发工作后执行，不可跳过。

1. **自动变更检测**
   - AI 分析修改的文件
   - 自动识别受影响的模块/功能/API
   - 生成全面的变更记录

2. **强制文档生成**
   - 所有受影响的文档必须更新
   - 变更记录必须生成
   - 所有文档更新前不能完成

3. **工作流完成**
   - 完成工作流会话
   - 最终确定所有文档
   - 启用审计跟踪

**用法：**
```
/docs-record [描述]
```

**示例：**
```bash
# 记录功能变更
/docs-record "添加用户登录功能"

# 记录 Bug 修复
/docs-record "修复认证错误"

# 记录重构
/docs-record "重构数据库层"
```

**强制机制：**
- 任何代码变更后必须生成文档
- 所有受影响的文档必须更新
- 变更记录必须完整
- 没有文档不能完成工作流

**生成内容：**
- `.ai-context/changelog/` 中的变更记录
- 更新的模块文档
- 更新的功能文档
- 更新的 API 文档
- 决策记录（如适用）

**输出：**
- `.ai-context/changelog/` 中的变更记录
- `.ai-context/modules/` 中更新的文档
- `.ai-context/features/` 中更新的文档
- `.ai-context/api/` 中更新的文档
- `.ai-context/decisions/` 中的决策记录

## 文档结构

### 供 AI 消费

```
.ai-context/
├── README.md                    # 项目概览和导航
├── architecture.md              # 系统架构
├── guidelines/                  # 编码标准和规范
│   ├── README.md                # 规范索引
│   ├── code-style.md            # 代码风格和格式
│   ├── naming-conventions.md    # 命名规范
│   └── git-workflow.md          # Git 工作流和分支
├── modules/                     # 模块文档
│   ├── [module-name]/           # 模块详情
│   │   └── README.md            # 模块摘要
├── features/                    # 功能文档
│   ├── README.md                # 功能索引和概览
│   └── [feature-name]/          # 每个功能一个文件夹
│       ├── README.md            # 功能摘要
│       ├── requirements.md      # 需求和规格
│       ├── design.md            # 设计决策
│       └── implementation.md    # 实现详情
├── api/                         # API 文档
│   ├── README.md                # API 索引和概览
│   └── [api-name]/              # 每个 API 一个文件夹
│       ├── README.md            # API 概览
│       ├── endpoints.md         # 端点文档
│       ├── schemas.md           # 数据模式
│       └── examples.md          # 使用示例
├── changelog/                   # 变更记录
│   └── YYYY-MM-DD-[type]-[name].md  # 单独的变更记录
└── decisions/                   # 决策日志
    └── YYYY-MM-DD-[decision].md     # 单独的决策记录
```

### AI 可读性标准

1. **清晰层次**
   - 一致的标题级别
   - 逻辑组织
   - 易于导航

2. **结构化内容**
   - YAML frontmatter 元数据
   - 一致的章节格式
   - 机器可解析结构

3. **明确关系**
   - 清晰的依赖文档
   - 文档间交叉引用
   - 影响链文档化

4. **可操作信息**
   - 清晰说明
   - 示例和用法
   - 故障排除指南

## 工作流

### 初始设置工作流

1. 运行 `/docs-scan` 扫描代码库
2. 审查生成的文档
3. 根据需要自定义模板
4. 建立文档标准

### 开发工作流

1. **开发前**
   ```
   /docs-read [要开发的功能]
   ```
   - 读取所有相关文档
   - 理解当前状态
   - 识别依赖和风险

2. **开发中**
   - 遵循文档标准
   - 编码时更新文档
   - 实时记录决策

3. **开发后**
   ```
   /docs-record [变更内容]
   ```
   - 记录所有变更
   - 更新受影响的文档
   - 创建变更记录

### Bug 修复工作流

1. **调查**
   ```
   /docs-read [bug 区域]
   ```
   - 读取相关文档
   - 理解当前行为
   - 识别根本原因

2. **修复实现**
   - 实现修复
   - 更新文档
   - 记录变更

3. **验证**
   ```
   /docs-record [bug 修复描述]
   ```
   - 记录修复
   - 记录测试结果
   - 更新影响分析

## 模板

### 模块文档模板
```markdown
---
module: [模块名]
version: [版本]
status: [active/deprecated/experimental]
dependencies: [依赖列表]
---

# [模块名]

## 概述
模块目的和职责的简要描述。

## 接口
此模块暴露的公共 API 和接口。

## 实现
关键实现细节和模式。

## 依赖
此模块依赖什么，以及什么依赖它。

## 使用示例
常见使用模式和示例。

## 故障排除
常见问题和解决方案。
```

### 变更记录模板
```markdown
---
date: [YYYY-MM-DD]
type: [feature/bugfix/refactor/docs]
modules: [受影响模块]
impact: [low/medium/high]
---

# [变更标题]

## 摘要
变更的简要描述。

## 动机
为什么进行此变更。

## 变更内容
- [ ] 代码变更
- [ ] 文档更新
- [ ] 测试添加/修改

## 影响分析
- 直接影响：[列表]
- 间接影响：[列表]
- 风险评估：[级别]

## 决策日志
- 决策：[决定了什么]
- 备选方案：[考虑了什么]
- 理由：[为什么选择此方案]

## 测试
- 运行的测试：[列表]
- 结果：[通过/失败]
- 覆盖率：[百分比]

## 回滚计划
如需要，如何回退。
```

## 最佳实践

1. **文档优先**
   - 尽可能先写文档再写代码
   - 实时保持文档更新
   - 永远不要让文档落后

2. **AI 优化**
   - 使用一致格式
   - 提供清晰结构
   - 包含明确关系

3. **完整记录**
   - 记录所有决策
   - 记录所有变更
   - 维护审计跟踪

4. **定期审查**
   - 定期审查文档
   - 更新陈旧信息
   - 提高文档质量

## 错误处理

### 缺失文档
- 提醒用户缺失的文档
- 提供创建文档的指导
- 主动生成初始文档

### 陈旧文档
- 检测过时信息
- 提醒用户更新
- 提供更新指导

### 不完整记录
- 验证变更记录
- 提醒缺失信息
- 提供完成指导

## 集成

### 与 Git 集成
- 提交前 hook 更新文档
- 提交后文档同步
- 分支特定文档

### 与 CI/CD 集成
- 文档验证
- 自动文档生成
- 变更记录验证

### 与项目管理集成
- 链接到 issue/工单
- 跟踪文档任务
- 监控文档覆盖率
